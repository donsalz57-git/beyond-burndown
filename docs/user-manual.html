1. Getting Started

Beyond Burndown is a Jira Cloud dashboard gadget that provides comprehensive project analysis including feasibility scoring, data quality checks, dependency modeling, and What-If scenario planning.

Installation

  1. Go to the Atlassian Marketplace
  2. Search for "Beyond Burndown" or "Feasibility Analyzer"
  3. Click Get it now and select your Jira site
  4. Confirm the installation

Adding to a Dashboard

  1. Navigate to any Jira dashboard
  2. Click Add gadget (or the + button)
  3. Search for "Feasibility" or "Beyond Burndown"
  4. Click Add gadget
  5. Configure the gadget with your JQL queries

Initial Configuration

After adding the gadget, you need to configure it:

  1. Click the gear icon on the gadget
  2. Enter a Demand JQL query for your work items
  3. Optionally set up Capacity (via JQL or manual entry)
  4. Click Save
Quick Start Tip
Try starting with a simple JQL like project = YOUR_PROJECT AND sprint in openSprints() to see your current sprint's feasibility.

2. Feasibility Tab

The Feasibility tab is the main view showing how realistic your planned work is given available capacity.

Feasibility Score

The feasibility score is a 0-100% measure of how achievable your plan is.

How It's Calculated

Score = (Coverage Ratio - Overload Penalty) Ã— 100

Coverage Ratio measures total capacity vs total demand:

Coverage Ratio = min(1, Total Capacity Hours / Total Demand Hours)
  • If you have more capacity than demand, ratio = 1.0 (capped)
  • If demand exceeds capacity, ratio decreases proportionally

Overload Penalty penalizes uneven distribution:

Overload Penalty = min(0.3, (Overloaded Days / Total Days) Ã— 0.5)
  • Even with enough total capacity, having many overloaded days reduces the score
  • Maximum penalty is 30% (0.3)

Score Ranges

Score Color Meaning Recommendation
80-100% Green Healthy Plan is realistic with buffer for uncertainty
50-79% Orange At Risk Consider reducing scope or adding capacity
0-49% Red Overloaded Plan is likely to fail without changes

Chart Elements

The feasibility chart shows several data series plotted over time:

Line Color Style Description
Period Capacity Green Solid bars Available capacity for each period
Period Demand Blue Solid bars Required work for each period
Cumulative Capacity Green Dashed line Running total of capacity over time
Cumulative Demand Blue Dashed line Running total of demand over time
Cumulative Time Spent Orange Dashed line Running total of logged work
Completion % Purple Solid line Time spent / original estimate (right Y-axis)

Overload Warnings

When demand exceeds capacity for a period, a red warning banner appears showing:

  • The date range of the overload
  • Number of days affected
  • Maximum hours over capacity

Period Selector

Choose how to aggregate data for different planning horizons:

3. Compliance Tab (Data Quality)

The Compliance tab identifies data quality issues in your Jira issues that could affect planning accuracy.

Violation Types

Missing Start Date

Issues without a start date cannot be properly scheduled. The analyzer assumes work starts immediately, which may not be accurate.

Fix: Add a start date to the issue or use the "Start date" custom field.

Missing Due Date

Issues without a due date are assumed to have a 14-day window, which may misrepresent actual deadlines.

Fix: Set a realistic due date for the issue.

Missing Estimate

Issues without time estimates cannot contribute to capacity calculations. They're invisible to feasibility analysis.

Fix: Add an original estimate (time tracking) to the issue.

Overdue

Issues past their due date that aren't marked as Done. These represent planning failures or outdated data.

Fix: Either complete the issue, update the due date, or mark it as blocked.

Done with Remaining Work

Issues marked as Done but still have remaining estimate. This suggests incomplete logging or incorrect status.

Fix: Clear the remaining estimate or verify the issue is actually complete.

Child After Parent

Sub-tasks or child stories with due dates after their parent Epic's due date. This creates impossible schedules.

Fix: Adjust either the child's due date or the parent's due date.

Dependency Conflict

A blocking issue finishes after the blocked issue is supposed to start. This creates impossible execution order.

Fix: Adjust dates so blockers finish before dependent work begins.

Severity Levels

Severity Color Meaning
Error Red Critical issue that will cause planning failures
Warning Orange Potential problem that may affect accuracy
Info Blue Minor issue or suggestion for improvement

4. Dependencies Tab

The Dependencies tab analyzes blocking relationships between issues to help you understand execution order and identify problems.

Dependency Graph

The dependency analysis builds a directed graph from Jira issue links:

Link Types Analyzed

  • Blocks / Is blocked by - Primary blocking relationships
  • Has to be done before / Has to be done after
  • Is required by / Requires

Metrics Calculated

Metric Description
Dependency Depth How many levels of blockers an issue has. Depth 0 = no blockers.
Root Issues Issues with no blockers - can start immediately.
Leaf Issues Issues that don't block anything - can finish last.
Topological Order Optimal execution sequence respecting all dependencies.

Circular Dependencies

A circular dependency occurs when issues form a cycle (A blocks B, B blocks C, C blocks A). These are impossible to execute and must be resolved.

Circular Dependency Detected
If circular dependencies are found, they're displayed prominently with the cycle path. You must break the cycle by removing one of the blocking links.

5. Report Tab

The Report tab generates professional status reports for stakeholder communication.

Report Contents

  • Executive Summary - High-level status with feasibility score
  • Key Metrics - Total demand, capacity, time spent, completion %
  • Risk Highlights - Top compliance violations and concerns
  • Overload Periods - Dates when demand exceeds capacity
  • Dependency Status - Circular dependencies and blocked chains

Export Options

Format Best For
Copy as Text Email, Slack, plain text documents
Copy as Markdown Confluence, GitHub, documentation systems

6. Team Tab

The Team tab shows workload distribution across team members.

Workload Analysis

For each assignee, the Team view shows:

  • Assigned Hours - Total estimated work assigned to them
  • Time Spent - Hours already logged
  • Remaining - Hours still to complete
  • Utilization - Percentage of their capacity used

Identifying Imbalances

  • Red - Over capacity (>100% utilization)
  • Orange - Near capacity (80-100%)
  • Green - Healthy (<80%)

Unassigned Work

Work without an assignee is shown separately. This helps identify gaps in task assignment.

7. Capacity Tab

The Capacity tab lets you configure team capacity without creating Jira issues.

Manual Capacity Entry

Instead of using capacity issues, you can directly specify each team member's availability:

  1. Go to the Capacity tab
  2. Select period type: Weekly or Monthly
  3. Click Add Team Member
  4. Enter their name and hours per period
  5. Optionally set start/end dates for temporary capacity
  6. Click Save

Capacity Calculation

8. What-If Scenarios

The What-If panel lets you model hypothetical changes and see their impact on feasibility before committing to them.

Opening What-If

Click the Show What-If button in the header to open the scenarios panel.

Capacity Scenarios

Model changes to team capacity:

Add Capacity (+)

Simulate adding resources:

  • +10% - Modest increase (overtime, temporary help)
  • +25% - Significant increase (new team member)
  • +50% - Major increase (multiple hires)
  • Custom - Enter any percentage

Remove Capacity (-)

Simulate losing resources:

  • -10% - Minor reduction (sick days, training)
  • -25% - Vacation or temporary leave
  • -50% - Major reduction (departure, reassignment)

Scope Scenarios

Model changes to planned work:

Remove Scope (-)

Simulate descoping:

  • -10% - Minor feature cuts
  • -25% - Significant descope
  • -50% - Major scope reduction

Add Scope (+)

Simulate scope creep or new requirements:

  • +10% - Minor additions
  • +25% - New feature request
  • +50% - Major scope increase

Deadline Scenarios

Model timeline changes:

  • +1 week - Extend deadline by 7 days
  • +2 weeks - Extend by 14 days
  • Custom - Specify any number of days

Combining Scenarios

You can activate multiple scenarios simultaneously to model complex changes:

Example: Finding the Sweet Spot

If your feasibility is 45% (red), try:

  2. Also remove -20% scope â†’ Score becomes 78%
  3. The combination gets you to a healthy state

Active Scenarios Indicator

When scenarios are active, the header shows "Viewing X scenarios" with the combined effect. The feasibility score updates in real-time to reflect the scenario impact.

9. Configuration

Access configuration by clicking the gear icon on the gadget.

Configuration Options

Setting Description Required
Demand JQL JQL query to select work items to analyze Yes
Capacity JQL JQL query for capacity-representing issues No*
Manual Capacity Use the Capacity tab instead of JQL No*

*Either Capacity JQL or Manual Capacity should be configured.

Required Issue Fields

For best results, issues should have:

  • Start Date - When work begins (custom field or sprint start)
  • Due Date - When work must complete
  • Original Estimate - Total planned hours
  • Remaining Estimate - Hours left to complete
  • Worklogs - Time logged against the issue

10. JQL Examples

Demand JQL

Current Sprint

project = MYPROJECT AND sprint in openSprints()

Specific Sprint

project = MYPROJECT AND sprint = "Sprint 42"

Date Range

project = MYPROJECT AND duedate >= 2026-01-01 AND duedate <= 2026-03-31

Excluding Done Issues

project = MYPROJECT AND sprint in openSprints() AND status != Done

Using Saved Filter

filter = 12345

Capacity JQL

Capacity Issues by Summary

project = MYPROJECT AND summary ~ "Capacity"

Capacity Issues by Label

project = MYPROJECT AND labels = "team-capacity"

Specific Team

project = MYPROJECT AND labels = "team-alpha-capacity"
Capacity Issue Pattern
A common pattern is to create one "Capacity" issue per sprint with:
  • Summary: "Sprint 42 - Team Capacity"
  • Start Date: Sprint start
  • Due Date: Sprint end
  • Original Estimate: Total team hours available

11. Troubleshooting

No Data Showing

Possible Causes
  • JQL query returns no issues - test it in Jira's issue search
  • Issues have no estimates - add Original Estimate values
  • Issues have no dates - add Start Date and Due Date
  • Filter permissions - ensure the filter is shared appropriately

Feasibility Score is 0%

This means no capacity was detected:

  • Check your Capacity JQL query
  • Verify capacity issues have Original Estimate set
  • Verify capacity issues have both Start Date and Due Date
  • Try using Manual Capacity in the Capacity tab

Completion Percentage Stuck at 0%

  • Issues need Original Estimates set (not just Remaining)
  • Time must be logged via Worklogs
  • Ensure worklogs are within the analysis date range

Chart Shows Wrong Date Range

The date range is calculated from:

  • Earliest start date across all issues
  • Latest due date across all issues
  • Plus a 5-day buffer on each end

Check that your issues have reasonable dates.

"Filter Not Found" Error

When using filter = XXXXX syntax:

  • The filter must exist and be accessible to you
  • Go to the filter and check sharing permissions
  • Share with "Any logged-in user" or your group

Performance Issues

If the gadget loads slowly:

  • Narrow your JQL to fewer issues
  • Use date range filters to limit the scope
  • Consider using saved filters for complex JQL
